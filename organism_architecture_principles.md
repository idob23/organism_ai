# Архитектурные принципы «Организм»

## Ключевая идея

«Организм» — это не монолитное приложение, а платформа оркестрации AI-агентов, которая адаптируется под конкретный бизнес через конфигурацию, а не через переписывание кода.

---

## 1. Разделение ядра и конфигурации

Всё, что делает «Организм», должно быть разделено на два слоя:

**Ядро (неизменное между клиентами):**
- Движок оркестрации агентов (кто, когда, в каком порядке действует)
- Система маршрутизации задач (определение какой агент обрабатывает запрос)
- Управление контекстом и памятью агентов
- Логирование, мониторинг, обработка ошибок
- Канало-агностический шлюз коммуникации (Telegram, CLI, web UI — через единый интерфейс)
- Механизм интеграции с внешними системами (MCP-клиент)

**Конфигурация (меняется от клиента к клиенту):**
- Набор активных агентов и их роли
- Справочники и словари предметной области (номенклатура, площадки, должности)
- Шаблоны документов (путевые листы, табели, заявки)
- Правила маршрутизации и эскалации
- Права доступа и роли пользователей
- Промпты и инструкции для каждого агента
- PERSONALITY.md — стиль общения, терминология, правила эскалации для конкретной артели

**Версионирование конфигурации:** конфигурации каждого клиента хранятся в Git. Любое изменение можно откатить. PromptVersionControl (уже реализован) расширяется на всю конфигурацию — промпты, справочники, правила маршрутизации.

**Критерий:** если для подключения новой артели нужно менять код ядра — архитектура неправильная. Должно хватать новой конфигурации.

---

## 2. Агенты как модульные единицы

Каждый агент — это отдельный изолированный модуль с чётким интерфейсом.

**Структура агента:**
- Имя и описание (что делает)
- Входные данные (что принимает)
- Выходные данные (что возвращает)
- Инструменты (к каким системам имеет доступ)
- Промпт (инструкции поведения)
- Ограничения (что НЕ должен делать)
- Метрики (как измерять качество его работы)

**Принцип:** добавление нового агента (например, «агент контроля ГСМ») — это создание конфигурационного файла и регистрация в системе, а не рефакторинг всего проекта.

**Межагентная коммуникация:** агенты общаются только через шину сообщений (Event Bus: asyncio Queue на MVP, Redis при масштабировании). Прямые вызовы между агентами запрещены. Это даёт:
- Замену одного агента без влияния на остальных
- Логирование всех межагентных коммуникаций
- Возможность распределить агентов по разным процессам/серверам в будущем

**Типы агентов по функции:**
- Агенты-исполнители (выполняют конкретную задачу: нормализация номенклатуры, формирование отчёта)
- Агенты-маршрутизаторы (определяют, кому передать задачу)
- Агенты-контролёры (проверяют результат работы других агентов) — *планируется, текущий Evaluator выполняет эту роль частично*
- Агент-диспетчер (точка входа, понимает запрос пользователя и запускает цепочку)

## 3. Интеграция с 1С как отдельный слой

1С — это сердце любой артели. Интеграция должна быть:

**Абстрагированная:** агенты не работают с 1С напрямую. Между ними — слой адаптера через MCP (Model Context Protocol), который переводит команды агентов в запросы к 1С и обратно. Если завтра артель перейдёт на другую версию 1С или добавит новую конфигурацию — меняется только MCP-сервер адаптера.

**Read-only по умолчанию:** любые операции чтения (поиск контрагентов, остатки ГСМ, реестр оборудования) доступны агентам свободно. Операции записи — только через черновики с обязательным подтверждением человека. Прямая запись в 1С без подтверждения запрещена на уровне архитектуры. Это не настройка, а жёсткое ограничение.

**Версионируемая:** разные артели используют разные конфигурации 1С. Адаптер должен поддерживать несколько версий схемы данных.

**Кэширующая:** справочники и нечасто меняющиеся данные кэшируются локально, чтобы не зависеть от скорости/доступности 1С-сервера (в удалённых площадках это критично). TTL кэша настраивается в конфигурации клиента.

## 4. Надёжность как приоритет №1

Для промышленного внедрения надёжность важнее функциональности. Лучше 5 агентов, которые работают безотказно, чем 20, которые ломаются.

**Принципы надёжности:**

- **Человек в контуре на критических операциях.** Агент предлагает — человек подтверждает. Особенно на этапе внедрения. Автоматизация увеличивается постепенно по мере роста доверия.
- **Graceful degradation.** Если LLM-провайдер недоступен или агент вернул ерунду — система не падает. Задача помечается как требующая ручной обработки, пользователь получает уведомление. Каждый компонент обёрнут в try/except — отказ одного не роняет остальные.
- **Повторяемость.** Одинаковый вход → одинаковый выход (насколько это возможно с LLM). Критические операции используют низкую temperature и структурированный вывод.
- **Логирование всего.** Каждый запрос к LLM, каждый ответ, каждое решение агента — записывается. Это и для отладки, и для доказательства результатов клиенту.
- **Таймауты и fallback.** Каждая операция имеет таймаут. Если агент не ответил за N секунд — срабатывает fallback-сценарий.

---

## 5. Работа в условиях плохой связи

Артели часто работают на удалённых площадках. Архитектура должна учитывать:

- **Асинхронность по умолчанию.** Задачи ставятся в очередь и обрабатываются когда есть связь. Пользователь не должен ждать ответа в реальном времени для некритических операций.
- **Локальный буфер.** Данные с площадок накапливаются локально и синхронизируются при появлении связи.
- **Минимальный трафик.** Telegram — лёгкий протокол, это плюс. Но агенты не должны отправлять огромные файлы или делать десятки API-вызовов на один запрос пользователя.
- **Офлайн-режим для базовых функций.** Просмотр справочников, шаблонов, последних данных — без интернета. Кэш 1С-данных работает автономно.

## 6. Метрики и доказательная база

С первого дня работы системы нужно собирать:

**Операционные метрики:**
- Время выполнения задачи (до и после AI)
- Количество ошибок (до и после)
- Процент задач, решённых без вмешательства человека
- Время простоя системы

**Бизнес-метрики:**
- Экономия рабочего времени в часах/рублях
- Снижение потерь от ошибок в документах
- Скорость обработки заявок

**Метрики качества AI:**
- Точность ответов агентов (процент принятых предложений)
- Количество откатов/исправлений
- Среднее количество итераций до правильного результата
- Success rate и quality_score из benchmark.py

**Техническая реализация:** метрики хранятся в PostgreSQL (та же БД, отдельные таблицы — task_memories, agent_reflections уже содержат базовые метрики). Агрегация — через SQL views. На фазе 1 не тянуть Prometheus/Grafana — достаточно /stats в Telegram и еженедельного автоматического отчёта (проактивный планировщик, Sprint 6). Дашборд — на фазе 2.

**Зачем:** это не просто мониторинг. Это sales-материал для следующей артели. «У артели X мы сократили обработку путевых листов на 70%» — это аргумент, который продаёт.

## 7. Безопасность и разграничение доступа

В артели работают разные люди с разным уровнем доступа:

- **Ролевая модель.** Начальник участка видит одно, кладовщик — другое, бухгалтер — третье. Агенты должны учитывать роль пользователя при выдаче информации.
- **Аудит действий.** Кто запросил, что получил, что изменил — всё записывается.
- **Защита данных.** Информация о добыче, зарплатах, остатках — конфиденциальна. Чувствительные данные обрабатываются через структурированные запросы, а не через свободный текст.
- **Осознанный риск: облачные LLM.** При использовании Claude API данные уходят на серверы Anthropic. Anthropic не использует данные API для обучения моделей. На фазе 1 это допустимый компромисс. План митигации: для особо чувствительных операций (зарплаты, объёмы добычи) — структурированные запросы без отправки сырых данных в LLM; на фазе 3 рассмотреть локальную модель для чувствительных операций.
- **API-ключи и доступы.** Централизованное управление, ротация, мониторинг использования.

## 8. Масштабирование: от одной артели к продукту

**Фаза 1 (сейчас): Пилот на своей артели**
- Максимальная связка с конкретными процессами
- Ручная настройка и отладка
- Сбор метрик и кейсов
- Допустимо «грязное» решение, если работает

**Фаза 2 (через 3-6 месяцев): Рефакторинг в продукт**
- Вычленение универсального ядра из пилота
- Создание системы конфигурации
- Документация для внедрения
- Первый внешний клиент (соседняя артель)

**Фаза 3 (6-12 месяцев): Платформа**
- Самостоятельное подключение новых клиентов (с минимальной помощью)
- Маркетплейс агентов (готовые модули под типовые задачи)
- Панель управления для администратора артели
- SaaS-модель или лицензирование

**Критически важно:** не перепрыгивать фазы. Фаза 1 — это фундамент. Без реально работающего пилота с цифрами всё остальное — фантазия.

## 9. Технический стек: принципы выбора

Не привязываться к конкретной LLM. Сегодня — Claude API, завтра может быть другая модель или локальная. Поэтому:

- **Абстракция LLM-провайдера.** Единый интерфейс LLMProvider для вызова моделей. Смена провайдера — замена одного модуля. Промпты оптимизированы под основную модель (Claude), но архитектура позволяет fallback на другую при недоступности.
- **Канало-агностический шлюз.** Telegram — основной интерфейс на фазе 1, но логика агентов не привязана к каналу. Подключение веб-панели, мобильного приложения или голосового интерфейса — это новый адаптер канала, не переписывание агентов.
- **Простые, проверенные технологии.** Для MVP на одну артель не нужен Kubernetes. PostgreSQL + pgvector, Python 3.11+, asyncio Queue — этого достаточно. Усложнение — потом, когда будет реальная нагрузка.
- **Docker для воспроизводимости.** Развёртывание на новом сервере — один docker-compose up.
- **MCP как стандарт интеграции.** Новая внешняя система = новый MCP-сервер, а не новый код в ядре. Organism AI — и MCP-клиент (подключает внешние инструменты), и MCP-сервер (выставляет свои возможности для других AI-систем).

## 10. Стратегия обновлений и деплоя

Система работает на удалённых площадках. Обновление не должно требовать физического присутствия.

- **Обновление через docker-compose pull + restart.** Новая версия — это новый Docker-образ. Откат — указание на предыдущий тег.
- **Health check после обновления.** Автоматическая проверка работоспособности после перезапуска. Если health check не прошёл — автоматический откат на предыдущую версию.
- **Канареечные обновления.** При нескольких артелях — сначала обновляем одну, проверяем неделю, потом остальные.
- **Разделение данных и кода.** Volumes с данными (PostgreSQL, логи, конфигурация) не затрагиваются при обновлении кода. Миграции БД — идемпотентные (безопасно запускать повторно).

---

## 11. Автономность после переезда

Через 10 месяцев — переезд в Питер. Система должна работать без физического присутствия:

- **Удалённый доступ ко всем компонентам.** VPN или SSH-туннели до серверов артели.
- **Автоматический перезапуск.** Если сервис упал — systemd/docker restart поднимает его без ручного вмешательства.
- **Мониторинг и алерты.** Если что-то сломалось — уведомление в Telegram тебе лично. Проактивный планировщик (Sprint 6) берёт на себя мониторинг аномалий.
- **Контактное лицо на месте.** Один человек в артели, который может перезагрузить сервер или проверить физическое подключение. Его нужно найти и обучить заранее.
- **Документация для экстренных ситуаций.** Пошаговая инструкция «что делать если...» для человека без технических знаний.
